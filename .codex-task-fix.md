## Bug 1: Garbled plan text (hook file)

### Problem
In the `readSseStream` callback inside `sendMission`, BOTH assistant and thinking tokens are accumulated into `streamText`. The thinking tokens often contain fragments that repeat/overlap with the assistant text, creating garbled duplicated output.

### Fix
Only accumulate `assistant` tokens into `streamText`. Skip `thinking` tokens entirely — they add noise.

Find this code in the sendMission mutationFn, inside the readSseStream callback:

```ts
if (event.type === 'assistant' || event.type === 'thinking') {
  setStreamText((current) => current + event.text)

  if (event.type === 'assistant' && !seenToolCallRef.current) {
    setPlanText((current) => current + event.text)
  }
```

Change it to:

```ts
if (event.type === 'assistant') {
  setStreamText((current) => current + event.text)

  if (!seenToolCallRef.current) {
    setPlanText((current) => current + event.text)
  }
```

Remove `thinking` from this block entirely. Thinking tokens should not be accumulated.

Also move the worker label extraction INSIDE the assistant-only block (labels won't appear in thinking tokens):

The full replacement for the event handling section should be:

```ts
if (event.type === 'assistant') {
  setStreamText((current) => current + event.text)

  if (!seenToolCallRef.current) {
    setPlanText((current) => current + event.text)
  }

  const labels = extractWorkerLabels(event.text)
  if (labels.length > 0) {
    setMissionWorkerLabels((current) => {
      const next = new Set(current)
      let changed = false
      for (const label of labels) {
        if (!next.has(label)) {
          next.add(label)
          changed = true
        }
      }
      return changed ? next : current
    })
  }
}
```

## Bug 2: Output preview path not found (conductor.tsx)

### Problem
The `extractProjectPath` function and `buildProjectPathCandidates` can't find the actual worker output because workers create directories with timestamps like `/tmp/dispatch-openclaw-notif-1774330061/` but the UI guesses paths without timestamps.

### Fix
In `conductor.tsx`, improve `extractProjectPath` to also search worker output text for ANY `/tmp/` path (not just `/tmp/dispatch-*`):

Find the `extractProjectPath` function and update it. The function should:

1. First try structured patterns (Created/Output/Wrote/etc.)
2. Then try `/tmp/dispatch-*` patterns (existing)
3. Then try ANY `/tmp/` path as a last resort

Add this as a third fallback in the function, after the existing dispatch pattern scan:

```ts
// Fallback: any /tmp/ path
const tmpMatches = text.match(/\/tmp\/[a-zA-Z0-9][^\s"')`\]>]+/g) ?? []
for (const raw of tmpMatches) {
  const cleaned = raw.replace(/[.,;:!?\-`]+$/, '')
  const normalized = cleaned.replace(/\/(index\.html|dist|build)\/?$/i, '')
  if (normalized.length > 5) return normalized
}
```

### Also fix: worker output in complete phase summary

In the `completeSummary` useMemo, also check worker outputs for the project path. Currently it only uses `completePhaseProjectPath` but that might miss paths that are only in worker output text.

In the complete phase rendering, the existing `completePhaseProjectPath` logic already checks tasks + workerOutputs + streamText. The real issue is the regex not matching timestamped paths. The fix above to `extractProjectPath` should handle this.
